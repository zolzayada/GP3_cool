# GP3_cool
Анализ рынка игр Steam

### Цель проекта
Провести аналитическое исследование рынка видеоигр на платформе Steam, чтобы определить, какие характеристики игры связаны с ее успехом у пользователей.

В качестве целевой переменной используется составная метрика **score** — взвешенная комбинация количества отзывов и рейтинга игры. Отзывы берутся как основа, потому что отражают реальный охват аудитории, рейтинг добавляется с меньшим весом для учета качества

### Данные

#### Скрапинг
Источником данных для скрапинга был сайт Steam (https://store.steampowered.com/).

Steam представляет собой крупнейший сервис дистрибуции игр. В рамках анализа были рассмотрены игры из магазина сервиса, так как на платформе представлено множество различных характеристик, полезных для анализа.

Мы выбрали именно Steam, потому что тут в структурированной форме и открытой содержится много различной информации об играх, присутствуют отзывы и оценки от большого количества пользователей, что в совокупности позволяет сделать содержательныц анализ.

Для сбора данных использовался веб-скрапинг с помощью:

* библиотеки httpx / requests для быстрых запросов и их оптимизации,
* BeautifulSoup для парсинга HTML,
* Selenium с бразуером Chrome (для обработки динамических страниц и страницы подтверждения возраста

## Пайплайн сбора данных:

1. Сбор ссылок на страницы игр (через каталог Steam)
2. Переход по каждой ссылке
3. Извлечение необходимых параметров со страницы игры
4. Обработка отдельных кейсов (проверка возраста, неполная загрузка страницы, fallback на Selenium при ошибках)

На платформе представлено много тысяч игр, но получить все данные напрямую путем парсинга не представлялось невозможным из-за динамической подгрузки контента, ограничений пагинации, возможных блокировок при частых запросах и ограничений по времени.

В рамках проекта был реализован сбор около 11 тысяч игр. По каждой игре собрали следующие признаки:  
link - ссылка на игру в steam
name - название игры
release_date - дата выпуска игры
developers - список разработчиков игры
genre - список жанров игры
price - цена игры
discount - скидка на платформе steam
dlc - список дополнений к игре
tags - теги к игре
rating - рейтинг игре на Steam 
reviews_count - количество отзывов на Steam recent_rating - недавний рейтинг Steam
recent_reviews_count - количество недавних отзывов на Steam

#### Подход к сбору данных

После скрапинга Steam было принято решение именно обогатить собранные данные дополнительными признаками через API. Для каждой игры из Steam датасета делались запросы к RAWG по названию игры. И во время этого собирались их айди с RAWG, для дальнейшего использования в эндпоинтах

#### API
Источник данных — [RAWG Video Games Database API](https://rawg.io/apidocs).

Использовалось несколько официальных API с ключом, тк лимит был 20к запросов, чего конечно не хватало, тк для каждого эндпоинта нужно было по 11к строк после скраппинга. Данные собирались по 5 эндпоинтам:

1. **`/games`** — поиск игры по названию. Получены: `rawg_rating`, `reviews_count`, `playtime`, `platforms`, `stores`, `esrb_rating`, `metacritic`, `id_games`
2. **`/games/{id}/achievements`** — количество достижений в игре (`achievements_count`)
3. **`/games/{id}/additions`** — количество дополнений, DLC и изданий (`a_count`)
4. **`/games/{id}/development-team`** — количество людей в команде разработки (`dev_team_count`)
5. **`/games/{id}/game-series`** — количество игр в той же серии (`series_count`)

Для ускорения сбора использовался `ThreadPoolExecutor` с 5 потоками. После первого эндпоинта данные собирались почти параллельно несколькими участниками команды по разным эндпоинтам, после чего объединялись по `id_games`

## Верификация качества мэтчинга
Поскольку игры сопоставлялись по названию (Steam - RAWG), часть совпадений могла быть некорректной(пробежались глазами, и некоторые случаи были такие). Для проверки качества мэтчинга использовалась библиотека rapidfuzz:

Названия нормализовались(нижний регистр, удаление спецсимволов и т д)
Считались две метрики схожести: fuzz.ratio и fuzz.token_sort_ratio
Каждая пара получала статус: good/check/bad

И по итогу строки со статусом bad были удалены из финального датасета как некорректные совпадения

### Пропущенные значения

В результате проведенного анализа было выявлено большое количество пропущенных значений в датафрейме. На основе анализа доли пропусков по признакам было принято решение удалить столбцы, в которых доля пропущенных значений превышала 50%, потому что они не несли достаточной аналитической ценности и могли исказить результаты последующего исследования, а именно metacritic, esrb_rating, recent_rating, recent_reviews_count, rating_steam, reviews_count_x .
В признаках, где было маленькое количество пропусков, то есть менее 1.5%, были удалены строки (discount, series_count, price, platforms, release_date, developers, dev_team_count, dlc, name). Их удаление не оказало значимого влияния на итоговый анализ
Оставшиеся пропуски были обработаны с учётом их смысла. В признаке a_count пропуски были заменены на 0, что говорит об отсутствии дополнительного контента у игры. В признаке stores пропущенные значения заменили на пустые списки, что говорит об отсутствии информации о платформах распространения.
После обработки пропусков была проведена проверка на наличие дубликатов. Обнаруженные дубликаты представляли собой полностью совпадающие строки и не содержали никакой новой информации, поэтому были удалены из датасета.

### Ограничения проекта

В процессе сбора данных возникли некоторые ограничения. Чтобы код не падал при ошибках загрузки страниц или парсинга, пропущенные значения заполнялись как None, из-за чего в датасете появились пропуски. Также при объединении данных часть игр могла потеряться из за различий в написании названий, хотя мы старались минимизировать это за счет очистки и проверки данных. Кроме того, из-за особенностей веб-скрапинга(динамическая загрузка страниц, различия в тегах html) не все данные удалось собрать полностью, и итоговая выборка может быть неполной.

### Feature Engeneering

Рейтинг сам по себе не до конца отражает успешность игры, так как не учитывает количество отзывов. Поэтому мы ввели новую переменную score, которая объединяет оба показателя.

Рейтинг был нормирован к диапазону значений 0–1, а количество отзывов логарифмировано (из-за сильного разброса) и также нормировано. Итоговый score считается как взвешенная сумма: 0.65 для отзывов и 0.35 для рейтинга.

Таким образом, метрика учитывает и популярность, и качество игры
